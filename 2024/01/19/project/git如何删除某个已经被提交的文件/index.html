git如何删除某个已经被提交的文件

发表于2024-01-19|更新于2024-01-21|project
|字数总计:1.4k|阅读时长:4分钟|阅读量:

前言

今天git push的时候一直上传不上去, 提醒我gitlab拒绝了我的请求, 我首先黑人问号, 然后想到了我提前不小心把我那个1G的镜像也提交了,所以有了这次的经历,我这次的经历还是比较曲折的, 首先, 那个要删除的文件本身并不是在我当前提交下的, 它是我提交了三个分支以后发现的问题, 导致我无法直接git checkout切回那个分支将它删除,同时也无法使用git rm --cached 文件名将它删除, 同时我错误的提问导致了gpt给了我个错误的命令git revert ,导致我删除了那个提交下的全部文件, 同时, 在我看到结果不对时 , 我错误的理解git revert的作用, 使用了错误的命令git reset 提交记录到了前面的提交,这大致就是我一整个经过,下面是我解决这个问题的过程

删除经过

首先, 因为我印象中廖雪峰讲过, git log的提交记录虽然会因为git reset而变得没有后面的提交, 但是实际上,它是不会删除后面的提交的, 只要我知道了后面提交的hash值, 则可以回到后面的分支, 由此复原到了git revert阶段

然后我通过询问gpt , 知道了git revert命令本身是可以通过继续git revert前面git revert命令生成的hash值来复原结果的, 所以我git revert了前面的hash值, 还原到了没有使用git revert阶段

然后, 我询问gpt , 我应该如何删除所有历史提交记录中的某个文件,他给了我这个代码git filter-branch --index-filter "git rm --cached --ignore-unmatch myfile.txt" HEAD, 终于, 我完成了我的需求

git revert的工作原理

git revert 命令在 Git 中用于“还原”之前的提交所做的更改。然而，这并不意味着删除或撤销之前的提交。相反，git revert 会创建一个新的提交，这个新提交包含了撤销之前指定提交更改的内容。这样做的结果是，之前的提交在项目历史中仍然存在，但是它的影响被新的提交所“抵消”。

这是 git revert 做的事情的具体步骤：

  1. 选择提交：当你运行 git revert [commit-hash] 时，你需要指定要还原的提交的哈希值。
  2. 生成反向更改：Git 会查看你指定的提交所做的更改，并生成一个新的”反向更改”集。这些更改与原始提交相反，用于取消原始提交的效果。
  3. 创建新提交：Git 将这些反向更改作为一个新的提交应用到当前分支上。这个新提交是原始提交的直接对立面。
  4. 保留历史：原始提交仍然存在于历史中，因为 git revert 并不修改既有的历史。
  5. 不影响分支：git revert 不会更改分支的引用。这意味着，与 git reset 不同，git revert 不会移动分支指针。

使用 git revert 的好处是它是一种安全的操作，因为它不会改变项目的历史。这使得它特别适合团队环境，因为它不会影响其他团队成员的历史。

例如，如果你在公共分支上错误地提交了一个更改并推送到了远程仓库，使用 git revert 可以创建一个新提交来抵消这个更改，而不需要重写项目历史或使用强制推送（git push --force），后者可能会导致其他团队成员遇到问题。

坑爹之处

当您执行 git revert 命令针对第四次提交时，Git 会创建一个新的提交，这个新提交包含了撤销第四次提交所做更改的内容。这个操作不会直接修改第四次提交本身，也不会改变第五次或第六次提交的内容。

然而，由于第四次提交的更改被新提交所抵消，它可能会间接影响第五次和第六次提交，具体情况取决于这些提交之间的关系。如果第五次和第六次提交与第四次提交没有修改同一部分代码，则它们不会受到直接影响。但如果第五次和第六次提交是在第四次提交所做更改的基础上进行的，那么撤销第四次提交可能会导致合并冲突，因为现在的项目状态与它们的基础不再一致。

在这种情况下，可能需要手动解决合并冲突。如果第五次和第六次提交依赖于第四次提交所引入的代码，那么撤销第四次提交可能会导致这些后续提交的代码不再有效或出现错误。

因此，尽管 git revert 不会改变现有提交的历史，但如果有后续提交依赖于被还原的提交所做的更改，这可能会引入复杂性。在执行 git revert 之后，您应该运行测试并仔细检查项目的状态，确保一切都按预期工作。

在团队环境中，如果您需要撤销一个公共分支上的提交，最好是与团队沟通更改，确保每个人都了解需要进行的调整。
